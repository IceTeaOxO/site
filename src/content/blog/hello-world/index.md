「我想做一個簡單心跳系統」開始 → 發現協定選擇背後其實是「通用性 vs 效能」的取捨 → 心跳語意設計帶出「liveness vs status」的分離思考 → receiver 主動偵測帶出「不能信任沉默」的可靠性設計原則 → 最後用 schema 註冊的討論收尾，談「過早抽象化」的陷阱與 YAGNI 原則的實務應用


1. 協定選擇的取捨：UDP vs TCP vs HTTP
這是最適合開場的技術討論。可以寫成「為什麼最後選 HTTP」的決策過程，而不只是結論：

UDP 的直覺吸引力：心跳本質上是「即使丟失，下一次很快會再來」，看似很適合無連線、輕量的 UDP
但「通用架構」改變了取捨的天平：當目標不是「单一場景優化」而是「給不同團隊/不同語言使用」時，可除錯性、防火牆穿透性、生態工具鏈的成熟度，重要性會超過協定本身的效能優勢
可以延伸討論：效能 vs 可維護性/可採用性的權衡，在基礎設施型工具（infra tooling）裡常常後者更重要，因為使用者的學習成本和除錯成本才是長期決定它會不會被棄用的關鍵

2. 心跳語意的兩種模式：被動輪詢 vs 主動推送的混合設計
你的設計裡其實藏了一個很值得寫的細節：「定期心跳」跟「狀態變更立即上報」這兩種語意要怎麼共存。

純定期心跳：簡單，但 error 發生到被發現之間最多會有一個 interval 的延遲
純事件驅動：即時，但如果 sender 程式整個卡死/掛掉，就完全不會有任何訊號
你的方案：兩者疊加——report() 立即送 + 背景心跳仍持續送「最後一次狀態」

這個設計模式其實很像分散式系統裡 liveness（活著嗎）vs status（狀態是什麼）兩個正交概念被疊在同一條訊息流裡，可以討論「為什麼要分開思考這兩種語意，又為什麼最後選擇合併傳輸」。
3. Receiver 端主動逾時偵測 —— "不能只信任 sender"
這是個值得展開的可靠性設計原則：

任何依賴對方「主動告知」的監控系統，都有一個天然漏洞：如果對方掛了，它連「我掛了」都說不出來
所以正確的責任歸屬應該倒過來：receiver 才是「判斷誰活著」的權威，而不是 sender
這呼應了分散式系統裡常見的設計哲學：「沉默」本身也是一種訊號，需要主動偵測，而不是等待

可以對比常見的「假監控」陷阱：只做 webhook/event-driven 通知，卻沒有處理「完全沒收到任何東西」的情境。
4. Schema 註冊：在「通用」與「簡單」之間找平衡點
這段討論其實展現了一個很常見的架構抉擇困境，非常適合寫成文章的核心：

通用化的誘惑：「我想要這個系統能用在所有情境」——這個念頭很自然，但過早抽象化的代價是複雜度前置
我們最後給的建議是「先寫死、先驗證流程，等真的遇到需求才加上 Schema 註冊層」——這其實是 YAGNI（You Aren't Gonna Need It）原則的具體案例
可以延伸寫：怎麼判斷「現在該不該做通用化」？一個簡單的判斷標準是：「如果你還沒有第二個真實使用情境，你寫的『通用』其實只是『猜測』」

5. Schema 註冊機制本身的設計：用「宣告」取代「硬編碼」
如果文章要更深入技術細節，這段可以展開成獨立小節：

核心思路是把「資料的形狀」變成可以動態傳遞的 metadata（fields + display_hint），而不是寫死在程式碼裡
這跟資料庫的 schema-on-write vs schema-on-read 概念有點呼應，可以做類比
display_hint 這個設計決定本身也值得拿出來討論：為什麼要把「顯示邏輯的線索」也放進 schema，而不是讓前端自己決定怎麼畫？——這是在「後端控制呈現」與「前端自主呈現」之間的取捨，通常的考量是：如果你想讓「新增一個服務」這件事不需要改前端程式碼，就必須把這類 hint 往後端的 schema 裡推

6. SDK 設計上的「非侵入性」原則
這個點在討論中沒有展開但其實隱含在程式碼設計裡，值得單獨寫一段：

reporter.start() 背景執行、不阻塞主程式——任何監控/可觀測性（observability）工具都有一個共同的設計鐵律：監控工具本身絕對不能成為被監控系統的故障點
可以延伸討論：重試機制要「有限次數」而不是無限重試，這也是同一個原則的延伸——監控代碼絕對不能因為自己重試太兇而拖垮宿主程式的效能
